## Notes — 4.8 / SR-1 read (price-free)

**Methodology guardrail (locked 2026-06-20).** The market gap is a **screening
signal, never an optimization target.** We calibrate to *correctness* (accounting /
finance defects whose right answer is independent of price) and drive the
**price-free** columns — terminal-PV share, negative-FCF years, negative intrinsic —
toward zero. A correct engine is *expected* to still disagree with the market on
genuinely mispriced names; a gap collapsing to ~0 across the whole basket would
signal **overfitting to price**, not accuracy. See `cmd/accuracy/README.md`
§"The gap is a signal, not a target" and `docs/FEEDBACK-LOG.md` (2026-06-20).

**Caveats on the absolute numbers.** Captured with config-fallback macro (no
`FRED_API_KEY` → static 10Y = 4.5%), and prices are 2026-06-20 while the 4.7
baseline used 2026-06-06. So the mean-gap delta vs 4.7 mixes the engine change with
~2 weeks of price drift — read the engine via the price-free columns below.

**SR-1 (4.7 → 4.8) is value-neutral on the basket.** Intrinsic values are unchanged
to <1% (AAPL 188.80→191.08, AMD 112.57→112.85, KO 15.74→15.73, MSFT 345.25→345.80,
NVDA 169.50→170.28, JPM 324.61→324.02; EQIX/F/MXL/PLD identical). The structural flag
rollup is **identical** to 4.7 (TERMINAL_DOMINANCE 6, NEG_FCF_YEARS 1,
MODEL_DIVERGENCE 1). SR-1's new parser extractions (R&D / COGS / effective-tax-rate,
commit `a7626de`) don't move the DCF because R&D already sits inside reported
operating income — they matter for *classification* (bucket D), not value. The
mean-gap drift (53.2% → 53.8%) and "9/10 below" (JPM slipped +3.9% → −0.4%) are
**price moves**, not engine.

**One real 4.8 change — data-quality grades dropped** (AMD/AAPL/MXL → F, MSFT → D,
NVDA → C, EQIX → B; ~all were A at 4.7). Cause: SR-1's B1 "revive the configured flag
system" + the disclosed quality-score delta (commit `e3d7311`). The grade now tracks
warning volume, which on these names is dominated by **valuation diagnostics**
(`terminal_dominance`, `reinvestment_model`, `operating_income_base=ANNUAL_FY`) rather
than raw data gaps — so a low grade here is not necessarily a data-quality problem.
Candidate follow-up: does `data_quality_grade` conflate model diagnostics with data
quality? (Out of scope for A-0, which only measures.)

**Frontier unchanged.** The systematic below-market skew + terminal dominance (80–89%
on the DCF names) is the same picture as 4.7. The lever remains **VAL-1**
(archetype-aware horizon + terminal normalization), not SR-1.
